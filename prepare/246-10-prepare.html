An Introduction to Logging

The are two main ways that we attempt to track down bugs in our code. One approach is to use the debugger. The debugger is a powerful tool that lets us set breakpoints and walk through our code line-by-line, examining the state of variable at any given point in time. Learning to effectively use a debugger is a critical skill for every software engineer.

Another tool or approach to identifying what our programs are doing, especially when they arrive in an error state is to use logging. We are likely all familiar with the idea of sprinkling console messages throughout our programs, in an attempt to output the state of variables or the order of execution at various points in the process. While this process of console messages can be helpful, it has some inherent problems. One problem is that there is no priority level among the messages. We may want very detailed messages for one section of code, but afterward want to focus solely on the critical failures, and console messages make it very difficult to filter in a meaningful way.

Another potential problem with console outputs is that on a server, for example, a critical failure may occur, and a message may be displayed to some console output that no one will ever see.

A good logging framework builds on the power of outputting important data, but formalizes the process to overcome some of these shortcomings. For example, log messages are often tagged in some way to indicate the module or class to which they belong, and are given a priority level, each of which can be filtered on later. Further, many logging framework are flexible enough to allow a user to easily change, via a configuration file, which messages should appear on the console, which should be saved to files, and how they should be formatted.

There are several pre-built and well-used logging frameworks for Java and for almost any other language you are programming in. You should become familiar with them and try to use them in lieu of console messages for all the reasons stated above. This is why you may have noticed that in Netbeans, when you have it auto generate a try/catch block, it produces the lines:

 

} catch (Exception ex) {
  Logger.getLogger(MyClass.class.getName()).log(Level.SEVERE, null, ex);
}

This is using the native Java logging API to log your message with a tag of the class name, a severe priority level, and the details of the exception. If you choose not to do anything else, it will display the message to the console, but it is prepared to do much more.

Android has it's own logging system designed for its specific context. Because the apps will be running on end users mobile devices in production, the logging framework is not designed to log messages to the file system, but rather is focused on the debugging process.
