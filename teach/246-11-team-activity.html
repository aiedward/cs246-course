Team Activity - Javadocs

Introduction

Before meeting with your team, make sure to go over the Team Activity Preparation - Javadocs so that your time together can be as efficient as possible.

This activity will help you become more familiar with Javadocs and how they should be used with your project.

As you saw in your reading of the preparation material, Javadocs provide a way for us to annotate our methods and variables in a specific consistent manner so that documentation can automatically be generated, and IDE's can offer intellisense and code helps.

By using the correct Javadocs comments in our code, we can easily generate documentation that looks and behaves just like the standard documentation (e.g., http://docs.oracle.com/javase/7/docs/api/index.html?java/io/File.html ).

Instructions

  1. Generate the HTML javadocs for your project to see what is produced. This can be done via the command line "javadocs" utility, or through an IDE. In Android Studio, this can be done by clicking Tools -> Generate Javadocs, then make sure to select an output directory where you want the documentation to be put.

  2. In the box for "Other command line arguments" you need to specify the location of your Android SDK (JAR file), like so:

    -bootclasspath C:\Users\nathanael\AppData\Local\Android\sdk\platforms\android-21\android.jar

  3. In a file explorer, browse to that directory to see what was created. Look through the directory and its subdirectories to see just how many things were created for you.

  4. Next, in a browser, open the index.html file in the root of the output directory. This should display a web page for the root of your project with links to your different classes and methods. Click on some of these links to get familiar with the navigation.

  5. Identify a method in your code that has non-trivial parameters, but does not have any information in the generated documentation.

  6. Return to your Java code and find this method.

  7. Place the proper Javadoc comment block before the method. Make sure to give a description for the method, each parameter, and the return value (if applicable).

  8. Regenerate the html Javadocs, and navigate to this method again. Your new comments should be displayed next to the method and the parameters.

  9. Make sure that each member of the team can generate the Javadocs and view the comments that you added.

  10. Your final task is to go through the remainder of your project and fill in the proper Javadoc comments. Please note that according to our style guide this should be at a minimum, every public class and every public or protected method in those classes (with a few exceptions specifically noted), plus other classes/methods that are not trivial.

  11. You may divide up the task of adding the Javadoc comments among teammates, but make sure that all the Javadocs comments have been written before you finish your meeting, and all of your code is committed and pushed to GitHub.
